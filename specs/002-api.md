# API

## Objetivo

Definir a API inicial do SensorHub, responsável por expor operações de cadastro, consulta e gerenciamento dos domínios principais do produto.

A API deve ser implementada em Java 25 usando Spring Boot, JPA e Flyway. O Flyway da própria API será responsável por criar e evoluir toda a estrutura da base de dados PostgreSQL.

## Escopo

Incluído nesta spec:

- Estrutura inicial da API em Java 25 com Spring Boot.
- Persistência com JPA.
- Migrations com Flyway dentro da API.
- Dados iniciais inseridos por Flyway para desenvolvimento local.
- Tabelas iniciais para `User`, `Device`, `Environment` e `Measurement`.
- Chaves primárias UUID geradas pelo banco.
- Relacionamentos com chaves estrangeiras entre os domínios.
- Endpoints CRUD iniciais para `User`, `Environment` e `Device`.
- Endpoints somente de consulta para `Measurement`.
- Testes automatizados para migrations, persistência, regras de negócio e contratos HTTP.

Fora deste escopo inicial:

- Autenticação e autorização.
- Multiusuário avançado.
- Versionamento público da API além do prefixo inicial.
- Filtros avançados e ordenação customizada fora das consultas temporais iniciais de medições.
- Conversão de unidades de medida.
- Worker MQTT real.
- Integração direta com o app Flutter.

## Comportamento esperado

A API deve permitir que clientes cadastrem, atualizem, removam e consultem registros dos seguintes domínios:

- usuários;
- ambientes;
- dispositivos.

Medições devem ser expostas pela API apenas para consulta. A criação de medições deve acontecer pelo worker MQTT, que consome mensagens publicadas pelo simulador Python ou por sensores físicos. A API não deve oferecer endpoints públicos para criar, atualizar ou remover medições.

A API deve persistir os dados no PostgreSQL respeitando as regras de integridade definidas pelas migrations Flyway.

Ao iniciar, a API deve executar migrations pendentes antes de atender requisições. A geração automática de schema pelo JPA não deve substituir as migrations. O JPA pode validar o schema existente.

## Dados e contratos

### Banco de dados

O schema inicial deve ser criado por migrations Flyway armazenadas dentro da API.

As tabelas iniciais devem representar os domínios definidos em `specs/001-product-overview.md`:

- `users`
- `environments`
- `devices`
- `measurements`

Todas as chaves primárias devem ser UUIDs gerados pelo banco.

### Dados iniciais

As migrations Flyway devem inserir dados iniciais mínimos para permitir desenvolvimento local e validação manual sem depender de cadastro prévio pela API.

Os dados iniciais devem ser criados de forma idempotente, usando restrições naturais como `users.email` e `devices.hardware_uuid` para evitar duplicidade quando a migration for reaplicada em ambientes recriados.

Usuário administrador inicial:

```json
{
  "name": "admin",
  "email": "admin@sensorhub.com"
}
```

Dispositivo inicial vinculado ao usuário administrador:

```json
{
  "hardwareUuid": "b0fee3a6-ae91-4265-9365-36f793f32f06",
  "userEmail": "admin@sensorhub.com",
  "environmentUuid": null,
  "status": "ACTIVATED",
  "name": "Test sensor"
}
```

O `uuid` interno do usuário e do dispositivo deve continuar sendo gerado pelo banco. A migration deve resolver o `user_uuid` do dispositivo a partir do usuário com email `admin@sensorhub.com`.

### User

Representa o usuário dono dos ambientes e dispositivos.

Campos esperados:

- `uuid`
- `name`
- `email`
- `createdAt`
- `updatedAt`

Relacionamentos:

- Um usuário pode possuir vários ambientes.
- Um usuário pode possuir vários dispositivos.

### Environment

Representa um ambiente monitorado.

Campos esperados:

- `uuid`
- `userUuid`
- `name`
- `createdAt`
- `updatedAt`

Relacionamentos:

- Um ambiente deve pertencer a um usuário.
- Um ambiente pode possuir vários dispositivos.

### Device

Representa um sensor físico ou simulado cadastrado por um usuário.

Campos esperados:

- `uuid`
- `userUuid`
- `hardwareUuid`
- `name`
- `environmentUuid`
- `status`
- `createdAt`
- `updatedAt`

Relacionamentos:

- Um dispositivo deve pertencer a um usuário.
- Um dispositivo pode estar associado a no máximo um ambiente.
- Um dispositivo pode não estar associado a nenhum ambiente.
- Um dispositivo pode possuir várias medições.

Identificadores:

- `uuid` é a chave primária interna do dispositivo, gerada pelo banco.
- `hardwareUuid` é o UUID físico vindo do firmware ou simulador.
- `hardwareUuid` deve ser informado no cadastro do dispositivo.

Status:

- `ACTIVATED`: dispositivo ativo para ingestão e consulta.
- `INACTIVATED`: dispositivo desativado administrativamente, preservando histórico e bloqueando novas atualizações de dados pelo fluxo de ingestão.

O status inicial padrão de um dispositivo criado pela API ou pelas migrations deve ser `ACTIVATED`.

Enquanto um dispositivo estiver `INACTIVATED`, o worker MQTT ou qualquer fluxo de ingestão deve ignorar novas leituras para esse dispositivo e não deve criar novas medições. Consultas podem continuar retornando histórico já persistido e devem sinalizar `freshnessStatus` como `INACTIVATED`.

`devices` deve armazenar somente dados cadastrais e status administrativo. A última comunicação do dispositivo não deve ser persistida em `devices`; ela deve ser calculada pela API a partir da maior data `receivedAt` das medições relacionadas ao dispositivo.

### Measurement

Representa uma leitura recebida de um sensor físico ou simulado.

Campos esperados:

- `uuid`
- `deviceUuid`
- `temperature`
- `temperatureUnit`
- `humidity`
- `humidityUnit`
- `measuredAt`
- `receivedAt`

Relacionamentos:

- Uma medição deve pertencer a um dispositivo.

## Relacionamentos e integridade

As migrations devem criar chaves estrangeiras para preservar os relacionamentos principais:

- `environments.user_uuid` referencia `users.uuid`.
- `devices.user_uuid` referencia `users.uuid`.
- `devices.environment_uuid` referencia `environments.uuid` e deve aceitar nulo.
- `measurements.device_uuid` referencia `devices.uuid`.

Quando `devices.environment_uuid` estiver preenchido, o ambiente referenciado deve pertencer ao mesmo usuário do dispositivo. Essa regra pode ser garantida por validação da aplicação no primeiro momento e reforçada no banco quando houver desenho adequado de constraints.

O UUID recebido do firmware ou simulador não deve ser usado como chave primária da tabela `devices`. Ele deve ser persistido em `devices.hardware_uuid` com restrição de unicidade. A tabela `measurements` deve referenciar o UUID interno do dispositivo em `measurements.device_uuid`.

## Endpoints iniciais

Todos os endpoints devem usar JSON para entrada e saída.

Prefixo inicial sugerido:

```text
/api/v1
```

### Users

- `POST /api/v1/users`: cria um usuário.
- `GET /api/v1/users`: lista usuários.
- `GET /api/v1/users/{uuid}`: consulta um usuário por UUID.
- `PUT /api/v1/users/{uuid}`: atualiza um usuário.
- `DELETE /api/v1/users/{uuid}`: remove um usuário.

### Environments

- `POST /api/v1/environments`: cria um ambiente.
- `GET /api/v1/environments`: lista ambientes.
- `GET /api/v1/environments/{uuid}`: consulta um ambiente por UUID.
- `PUT /api/v1/environments/{uuid}`: atualiza um ambiente.
- `DELETE /api/v1/environments/{uuid}`: remove um ambiente.

### Devices

- `POST /api/v1/devices`: cria um dispositivo.
- `GET /api/v1/devices`: lista dispositivos.
- `GET /api/v1/devices/{uuid}`: consulta um dispositivo por UUID.
- `PUT /api/v1/devices/{uuid}`: atualiza um dispositivo.
- `DELETE /api/v1/devices/{uuid}`: remove um dispositivo.

O cadastro de dispositivo deve receber o `hardwareUuid` vindo do firmware ou simulador:

```json
{
  "hardwareUuid": "2f4a7d8e-3b6a-4a5c-9f2b-8f4d0d8f3c21",
  "userUuid": "b52b7111-8d98-42b4-a5fb-8bc4c17dd681",
  "environmentUuid": null,
  "status": "ACTIVATED",
  "name": "Sensor da sala"
}
```

A resposta deve incluir o `uuid` interno gerado pelo banco:

```json
{
  "uuid": "d1fb2c1a-2e7d-49e2-9aa6-3b196f7d27a0",
  "hardwareUuid": "2f4a7d8e-3b6a-4a5c-9f2b-8f4d0d8f3c21",
  "userUuid": "b52b7111-8d98-42b4-a5fb-8bc4c17dd681",
  "environmentUuid": null,
  "status": "ACTIVATED",
  "name": "Sensor da sala"
}
```

### Measurements

- `GET /api/v1/measurements`: lista medições de forma paginada.
- `GET /api/v1/measurements/{uuid}`: consulta uma medição por UUID.
- `GET /api/v1/devices/{deviceUuid}/measurements`: lista medições de um dispositivo de forma paginada.
- `GET /api/v1/devices/{deviceUuid}/measurements/latest`: consulta a medição mais recente de um dispositivo usando `measuredAt` como critério.

O endpoint de listagem por dispositivo deve receber o UUID interno do dispositivo em `{deviceUuid}`.

Parâmetros de paginação:

- `page`: página solicitada, iniciando em `0`.
- `size`: quantidade de itens por página.

Parâmetros temporais:

- `from`: timestamp inicial inclusivo.
- `to`: timestamp final inclusivo.
- `timeField`: campo temporal usado no filtro, aceitando inicialmente `measuredAt` ou `receivedAt`.

Para buscar medições de um sensor durante um intervalo de tempo, o cliente deve usar o endpoint de medições por dispositivo com query parameters:

```text
GET /api/v1/devices/{deviceUuid}/measurements?from={from}&to={to}&timeField={timeField}&page={page}&size={size}
```

Query parameters:

- `from`: opcional, timestamp inicial do intervalo em formato ISO 8601. Quando informado, a consulta deve retornar medições com o campo temporal escolhido maior ou igual a esse valor.
- `to`: opcional, timestamp final do intervalo em formato ISO 8601. Quando informado, a consulta deve retornar medições com o campo temporal escolhido menor ou igual a esse valor.
- `timeField`: opcional, define qual timestamp será usado no filtro de intervalo. Valores aceitos: `measuredAt` e `receivedAt`. Valor padrão: `measuredAt`.
- `page`: opcional, página solicitada, iniciando em `0`. Valor padrão: `0`.
- `size`: opcional, quantidade de itens por página. Valor padrão inicial sugerido: `50`.

Quando `from` e `to` forem informados juntos, a API deve retornar apenas medições dentro do intervalo fechado. Quando apenas um deles for informado, a consulta deve aplicar somente o limite correspondente.

Exemplo:

```text
GET /api/v1/devices/d1fb2c1a-2e7d-49e2-9aa6-3b196f7d27a0/measurements?page=0&size=50&from=2026-06-01T00:00:00Z&to=2026-06-02T00:00:00Z&timeField=measuredAt
```

Quando `timeField` não for informado, a consulta deve usar `measuredAt`.

As respostas paginadas devem informar pelo menos:

- itens da página;
- página atual;
- tamanho da página;
- total de itens;
- total de páginas.

Esse endpoint deve retornar medições brutas, sem agregações, sem redução de pontos e sem cálculos estatísticos. Ele é adequado para auditoria, exportação, depuração, telas tabulares e consultas em que o cliente precisa navegar página por página pelos registros persistidos.

O endpoint de medição mais recente deve retornar a medição do dispositivo com maior valor de `measuredAt`.

Exemplo:

```text
GET /api/v1/devices/d1fb2c1a-2e7d-49e2-9aa6-3b196f7d27a0/measurements/latest
```

Se o dispositivo existir, mas ainda não possuir medições, a API deve retornar `204 No Content`.

### Contratos para o app mobile

O app mobile deve consumir contratos específicos da API para evitar processamento de domínio no cliente.

Os contratos voltados ao app devem separar dados com baixa frequência de atualização dos dados de medição, que mudam continuamente. Assim, a tela inicial pode carregar metadados de dispositivos e ambientes com menor frequência e fazer polling apenas dos valores variáveis.

#### Dashboard de dispositivos

- `GET /api/v1/users/{userUuid}/dashboard/devices`: lista os dados estáveis dos dispositivos de um usuário para montar os cards da tela inicial.

Esse endpoint deve retornar todos os dispositivos do usuário com metadados de identificação, ambiente associado e status administrativo. Ele não deve retornar leituras, `freshnessStatus` ou séries temporais.

Parâmetros opcionais:

- `environmentUuid`: filtra dispositivos por ambiente.
- `status`: filtra por status administrativo do dispositivo, aceitando `ACTIVATED` ou `INACTIVATED`.

Exemplo de resposta:

```json
[
  {
    "deviceUuid": "d1fb2c1a-2e7d-49e2-9aa6-3b196f7d27a0",
    "hardwareUuid": "2f4a7d8e-3b6a-4a5c-9f2b-8f4d0d8f3c21",
    "name": "Sensor da sala",
    "environmentUuid": "bf3c99bb-0ab1-44e4-a75a-1d8f3575e9a1",
    "environmentName": "Sala",
    "deviceStatus": "ACTIVATED"
  }
]
```

#### Leituras atuais do dashboard

- `GET /api/v1/users/{userUuid}/dashboard/measurements/latest`: lista os dados variáveis de medição para os cards da tela inicial.

Esse endpoint deve retornar a última medição disponível, última comunicação e status de atualização calculado pela API para cada dispositivo do usuário. Ele é o endpoint esperado para polling da home.

`lastSeenAt` na resposta é um campo calculado e deve ser preenchido com o maior `measurements.receivedAt` do dispositivo. O cálculo de `freshnessStatus` também deve usar `receivedAt`, não `measuredAt`, para evitar depender do relógio do sensor.

Quando um dispositivo ainda não possuir medições, o item deve continuar presente na resposta, com `freshnessStatus` igual a `NO_DATA`, `lastSeenAt` nulo e `latestMeasurement` nulo.

Parâmetros opcionais:

- `environmentUuid`: filtra dispositivos por ambiente.
- `deviceStatus`: filtra por status administrativo do dispositivo, aceitando `ACTIVATED` ou `INACTIVATED`.

Exemplo de resposta:

```json
[
  {
    "deviceUuid": "d1fb2c1a-2e7d-49e2-9aa6-3b196f7d27a0",
    "freshnessStatus": "ONLINE",
    "lastSeenAt": "2026-06-04T18:15:00Z",
    "latestMeasurement": {
      "temperature": 24.7,
      "temperatureUnit": "CELSIUS",
      "humidity": 58.2,
      "humidityUnit": "RELATIVE_PERCENT",
      "measuredAt": "2026-06-04T18:15:00Z"
    }
  }
]
```

`freshnessStatus` deve ser calculado pela API. Valores iniciais:

- `ONLINE`: dispositivo ativo com leitura recente.
- `OFFLINE`: dispositivo ativo, mas sem leitura dentro da janela configurada pela API.
- `NO_DATA`: dispositivo ativo sem medições.
- `INACTIVATED`: dispositivo inativado administrativamente.

A janela inicial para considerar um dispositivo `OFFLINE` deve ser configurável na API, com padrão sugerido de 1 minuto.

#### Overview de medições do sensor

- `GET /api/v1/devices/{deviceUuid}/measurements/overview?from={from}&to={to}&bucket={bucket}`: retorna dados variáveis de medição para a tela de detalhe de um sensor.

Esse endpoint deve retornar leitura atual, série temporal preparada para gráfico e overview estatístico do período. Metadados estáveis do dispositivo devem ser obtidos por `GET /api/v1/devices/{deviceUuid}` ou reaproveitados do dashboard de dispositivos.

`latestMeasurement` deve representar a medição mais recente do dispositivo usando `measuredAt`, independentemente do intervalo solicitado. `series` e `overview` devem respeitar `from`, `to` e `bucket`.

Parâmetros:

- `from`: timestamp inicial inclusivo em formato ISO 8601.
- `to`: timestamp final inclusivo em formato ISO 8601.
- `bucket`: granularidade desejada para a série temporal, como `raw`, `1m`, `5m`, `1h` ou `1d`.

Quando `bucket` for `raw`, a API pode retornar medições individuais respeitando limites de paginação ou limite máximo de pontos. Quando `bucket` representar intervalo, a API deve agregar ou reduzir os pontos no backend antes de responder.

Exemplo de resposta:

```json
{
  "deviceUuid": "d1fb2c1a-2e7d-49e2-9aa6-3b196f7d27a0",
  "freshnessStatus": "ONLINE",
  "lastSeenAt": "2026-06-04T18:15:00Z",
  "period": {
    "from": "2026-06-04T12:15:00Z",
    "to": "2026-06-04T18:15:00Z",
    "bucket": "5m"
  },
  "latestMeasurement": {
    "temperature": 24.7,
    "temperatureUnit": "CELSIUS",
    "humidity": 58.2,
    "humidityUnit": "RELATIVE_PERCENT",
    "measuredAt": "2026-06-04T18:15:00Z"
  },
  "series": [
    {
      "timestamp": "2026-06-04T18:10:00Z",
      "temperature": 24.5,
      "humidity": 59.1
    }
  ],
  "overview": {
    "temperatureMax": 26.8,
    "temperatureMaxAt": "2026-06-04T15:25:00Z",
    "temperatureMin": 22.9,
    "temperatureMinAt": "2026-06-04T12:40:00Z",
    "temperatureAverage": 24.6,
    "humidityMax": 63.4,
    "humidityMaxAt": "2026-06-04T13:10:00Z",
    "humidityMin": 52.8,
    "humidityMinAt": "2026-06-04T17:45:00Z",
    "humidityAverage": 58.7,
    "measurementCount": 73
  }
}
```

Overview deve ser calculado pela API, não pelo app mobile. Na primeira versão, o overview mínimo contém máxima, mínima e média de temperatura e umidade, com timestamps para máximas e mínimas.

#### Diferença entre listagem e overview

Os endpoints `GET /api/v1/devices/{deviceUuid}/measurements` e `GET /api/v1/devices/{deviceUuid}/measurements/overview` devem coexistir porque atendem casos de uso diferentes.

`GET /api/v1/devices/{deviceUuid}/measurements` é uma consulta de dados brutos. Ela deve preservar a paginação, retornar registros individuais e permitir inspeção fiel do histórico persistido. A API não deve calcular overview, não deve aplicar `bucket` e não deve reduzir pontos nesse endpoint.

`GET /api/v1/devices/{deviceUuid}/measurements/overview` é um contrato de leitura orientado à UI. Ele deve retornar, em uma única resposta, leitura atual, série temporal preparada para gráfico e overview estatístico do período. Esse endpoint evita que o app mobile precise carregar várias páginas, agregar pontos ou calcular máximas, mínimas e médias localmente.

O app mobile deve usar `measurements/overview` para gráficos e overview da tela de detalhe. O endpoint paginado `measurements` deve ficar reservado para funcionalidades futuras como histórico tabular, auditoria, exportação ou depuração.

## Regras de negócio

- Todo registro persistido deve possuir UUID como identificador primário gerado pelo banco.
- UUID de usuário deve ser único.
- Email de usuário deve ser único.
- UUID interno de dispositivo deve ser único.
- `hardwareUuid` de dispositivo deve ser obrigatório e único.
- O cadastro de dispositivo deve receber o `hardwareUuid` vindo do firmware ou simulador.
- O status de dispositivo deve aceitar inicialmente apenas `ACTIVATED` e `INACTIVATED`.
- Dispositivos `INACTIVATED` não devem receber novas medições pelo fluxo de ingestão.
- `lastSeenAt` deve ser exposto apenas em contratos de leitura que precisam de última comunicação e deve ser derivado de `measurements.receivedAt`.
- Dispositivos criados sem status explícito devem iniciar como `ACTIVATED`.
- Um dispositivo deve pertencer a um usuário.
- Um ambiente deve pertencer a um usuário.
- Associação de dispositivo a ambiente é opcional.
- Um dispositivo pode estar associado a no máximo um ambiente por vez.
- Quando um dispositivo estiver associado a um ambiente, ambos devem pertencer ao mesmo usuário.
- Uma medição deve estar vinculada a um dispositivo existente.
- Payloads vindos de sensores devem identificar o dispositivo por `hardwareUuid`; o worker deve resolver esse valor para `devices.uuid` antes de persistir a medição.
- A API pública não deve permitir criação, atualização ou remoção de medições.
- Medições devem ser criadas apenas por fluxos de ingestão controlados, inicialmente pelo worker MQTT.
- Consultas de medições devem ser paginadas.
- Consultas de medições por dispositivo devem exigir o UUID interno do dispositivo.
- Consultas temporais de medições devem aceitar filtros por `measuredAt` ou `receivedAt`.
- A consulta de medição mais recente de um dispositivo deve usar sempre `measuredAt` como critério de ordenação.
- A API deve calcular `freshnessStatus` para os contratos de leituras usados pelo app mobile.
- A API deve calcular overview histórico para a tela de detalhe do sensor.
- A API deve preparar séries temporais para gráficos quando o app solicitar `bucket` agregado.
- O app mobile não deve precisar calcular máximas, mínimas, médias, status de atualização ou agregações de série temporal.
- Temperatura e umidade devem ser numéricas.
- Temperatura deve armazenar a unidade de medida informada pelo sensor, inicialmente `CELSIUS`.
- Umidade deve armazenar a unidade de medida informada pelo sensor, inicialmente `RELATIVE_PERCENT`.
- `receivedAt` deve ser definido pelo sistema no momento da persistência da medição.
- `measuredAt` deve representar o timestamp informado pelo sensor ou simulador.

## Erros e casos limite

- Requisições com UUID inválido devem retornar erro de validação.
- Requisições para registros inexistentes devem retornar `404 Not Found`.
- Criações ou atualizações com campos obrigatórios ausentes devem retornar erro de validação.
- Tentativas de violar unicidade devem retornar erro de conflito.
- Tentativas de associar um dispositivo a ambiente inexistente devem retornar erro de validação ou `404 Not Found`.
- Tentativas de associar um dispositivo a ambiente de outro usuário devem ser rejeitadas.
- Tentativas de cadastrar mais de um dispositivo com o mesmo `hardwareUuid` devem retornar erro de conflito.
- Requisições de criação, atualização ou remoção de medições pela API pública devem retornar `405 Method Not Allowed` ou não possuir rota registrada.
- Consultas de medições para dispositivo inexistente devem retornar `404 Not Found`.
- Consulta de medição mais recente para dispositivo existente sem medições deve retornar `204 No Content`.
- Consultas temporais com `from` maior que `to` devem retornar erro de validação.
- Consultas temporais com `timeField` inválido devem retornar erro de validação.
- Consulta de dashboard de dispositivos para usuário inexistente deve retornar `404 Not Found`.
- Consulta de leituras atuais do dashboard para usuário inexistente deve retornar `404 Not Found`.
- Consulta de overview de medições para dispositivo inexistente deve retornar `404 Not Found`.
- Consulta de overview de medições com `from` maior que `to` deve retornar erro de validação.
- Consulta de overview de medições com `bucket` inválido deve retornar erro de validação.
- Overview de medições sem registros no período deve retornar `deviceUuid`, período, série vazia e overview nulo ou indisponível de forma explícita.
- Criações ou atualizações de dispositivo com status inválido devem retornar erro de validação.

## Critérios de aceite

- Existe uma aplicação Spring Boot em Java 25 para a API.
- A API usa JPA para persistência.
- A API possui Flyway configurado.
- As migrations da API criam as tabelas iniciais.
- As migrations da API inserem o usuário inicial `admin@sensorhub.com`.
- As migrations da API inserem um dispositivo inicial vinculado ao usuário administrador com `hardware_uuid` `b0fee3a6-ae91-4265-9365-36f793f32f06`.
- As tabelas usam UUID gerado pelo banco como chave primária.
- As tabelas possuem chaves estrangeiras para os relacionamentos definidos.
- A tabela `devices` possui `hardware_uuid` obrigatório e único.
- A tabela `devices` possui `status` obrigatório com valores permitidos `ACTIVATED` e `INACTIVATED`.
- O JPA não substitui as migrations como fonte de criação do schema.
- Existem endpoints CRUD para `User`, `Environment` e `Device`.
- Existem endpoints somente de consulta para `Measurement`.
- Existe endpoint paginado para listar medições de um dispositivo pelo UUID interno do dispositivo.
- Existe endpoint para consultar a medição mais recente de um dispositivo usando `measuredAt`.
- Consultas de medições aceitam filtros temporais por `measuredAt` e `receivedAt`.

## Testes

A implementação da API deve incluir testes automatizados compatíveis com a stack Spring Boot.

### Migrations e banco

- Testar que as migrations Flyway sobem o schema inicial com sucesso.
- Testar que as tabelas principais são criadas: `users`, `environments`, `devices` e `measurements`.
- Testar que as chaves primárias são UUIDs.
- Testar que as chaves estrangeiras principais existem e impedem registros órfãos.
- Testar unicidade de `users.email`.
- Testar unicidade de `devices.hardware_uuid`.
- Testar que a migration insere o usuário inicial `admin@sensorhub.com`.
- Testar que a migration insere o dispositivo inicial com `hardware_uuid` `b0fee3a6-ae91-4265-9365-36f793f32f06`, status `ACTIVATED` e vínculo ao usuário administrador.
- Testar que `devices.status` rejeita valores fora de `ACTIVATED` e `INACTIVATED`.

### Repositórios e persistência

- Testar criação e consulta de `User`.
- Testar criação e consulta de `Environment` vinculado a `User`.
- Testar criação e consulta de `Device` vinculado a `User`.
- Testar criação de `Device` sem `Environment`.
- Testar criação de `Device` com `Environment`.
- Testar criação e consulta de `Measurement` vinculada a `Device` por UUID interno.

### Regras de negócio

- Testar rejeição de dispositivo com `hardwareUuid` duplicado.
- Testar rejeição de dispositivo associado a ambiente inexistente.
- Testar rejeição de dispositivo associado a ambiente de outro usuário.
- Testar rejeição de medição para dispositivo inexistente.
- Testar que `receivedAt` é definido pelo sistema no fluxo de persistência de medição.
- Testar que consultas de medição mais recente usam `measuredAt`, não `receivedAt`.

### Endpoints HTTP

- Testar CRUD de `User`.
- Testar CRUD de `Environment`.
- Testar CRUD de `Device`.
- Testar que a API pública não expõe criação, atualização ou remoção de `Measurement`.
- Testar listagem paginada de medições.
- Testar listagem paginada de medições por dispositivo.
- Testar filtros temporais de medições por `measuredAt`.
- Testar filtros temporais de medições por `receivedAt`.
- Testar consulta da medição mais recente por dispositivo.
- Testar dashboard de dispositivos por usuário com dados estáveis.
- Testar leituras atuais do dashboard por usuário.
- Testar cálculo de `freshnessStatus` para dispositivos online, offline, sem dados e inativados.
- Testar overview de medições com série temporal e overview estatístico.
- Testar máximas, mínimas e médias de temperatura e umidade no período.

- Testar rejeição de `bucket` inválido no overview.
- Testar resposta `204 No Content` para dispositivo existente sem medições.

### Comando esperado

O projeto da API deve documentar um comando para executar a suíte de testes, por exemplo:

```text
./gradlew test
```

ou:

```text
./mvnw test
```

O comando final deve seguir a ferramenta de build escolhida na criação da API.

## Observações técnicas

- O Flyway deve ficar dentro da API nesta fase para reduzir complexidade operacional.
- O diretório esperado para migrations é `apps/api/src/main/resources/db/migration`.
- O schema deve ser evoluído por SQL versionado.
- A configuração recomendada para JPA é validar o schema gerado pelas migrations, não recriá-lo automaticamente.
- O Docker Compose deve incluir PostgreSQL e API quando a aplicação for criada.
- O worker MQTT deve ser criado depois da API, usando o schema gerado pelas migrations da API.
